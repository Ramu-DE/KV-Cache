# KV Cache — Chapter 10: Emerging Research (2024–2025)

## 1. The Research Landscape

KV cache research in 2024-2025 operates at three levels simultaneously:

```
LEVEL             WHAT IT OPTIMIZES         EXAMPLES
──────────────    ─────────────────────     ─────────────────────────────
TOKEN-LEVEL       Which tokens to keep      H2O, SnapKV, PyramidKV, DuoAttention
                  (eviction & selection)    (see Chapter 8)

MODEL-LEVEL       How K/V are computed      GQA, MQA, MLA, CLA, KVSharer
                  (architecture changes)    (reduce size at design time)

SYSTEM-LEVEL      How K/V are managed       PagedAttention, vAttention, SGLang,
                  (memory & scheduling)     DistServe, CacheGen, InfiniGen
                                            (see Chapter 9)
```

The most interesting 2024-2025 work combines levels — e.g., DuoAttention
operates at token-level but requires model-level head classification.

## 2. Flash Attention 3 — H100-Native Attention

**Paper:** arxiv:2407.08608 (Shah et al., 2024)  
**Year:** 2024

### Background: Flash Attention 1 and 2

Flash Attention (Dao et al., 2022) was the first IO-aware attention algorithm.
Instead of materializing the full N×N attention matrix in HBM (slow),
it tiles the computation and keeps intermediate results in SRAM (fast):

```
Standard attention:  Q,K,V → [write N×N scores to HBM] → [read back] → output
Flash Attention 1:   Q,K,V → [tiles in SRAM, never touch HBM for scores] → output
                     IO complexity: O(N²) → O(N²·d/M) where M = SRAM size
```

Flash Attention 2 (Dao, 2023) improved the algorithm further:
- Better parallelism across attention heads and sequence positions
- Reduced non-matrix-multiply FLOP overhead
- 2-4x faster than FA1

### Flash Attention 3 — Hopper-Specific

FA3 targets NVIDIA H100 (Hopper architecture) specifically, exploiting
hardware features unavailable on older GPUs:

**WGMMA (Warpgroup Matrix Multiply Accumulate):**
H100's new asynchronous matrix multiply units allow overlapping
computation and data movement — FA3 pipelines them:

```
FA2 (synchronous):
  Step 1: load Q tile from HBM → SRAM    (wait)
  Step 2: load K tile from HBM → SRAM    (wait)
  Step 3: compute QK^T                   (wait)
  Step 4: softmax                        (wait)
  Step 5: load V tile                    (wait)
  Step 6: compute AV output              (wait)

FA3 (asynchronous, pipelined):
  load Q tile ──┐
                ├── overlap
  compute QK^T ─┘
                load K tile ──┐
                              ├── overlap
                compute AV   ─┘
  ...
```

**TMA (Tensor Memory Accelerator):**
H100 has a dedicated hardware unit for async memory copies.
FA3 uses TMA to prefetch tiles while computing on already-loaded data.

### Performance

On H100 GPU (fp16):
- Flash Attention 3: 1.5-2.0x faster than Flash Attention 2
- Reaches 75% of H100's theoretical peak FLOP/s for attention
- Particularly impactful for prefill phase (long sequences)

**Note:** FA3 is compute-oriented — it speeds up the attention *computation*,
not the KV cache *storage/retrieval*. It is most relevant during prefill.
During memory-bound decode, the bottleneck is bandwidth, not computation.

*Source: arxiv:2407.08608*

## 3. CLA — Cross-Layer Attention (Deeper Treatment)

**Paper:** arxiv:2405.12981 (Brandon et al., 2024)  
**Year:** 2024

*(Brief mention in Chapter 4 — full treatment here)*

### The Within-Layer Baseline

GQA/MQA share K/V across **query heads within one layer**.
CLA extends this to share K/V across **transformer layers**:

```
Standard MHA:
  Layer 1: K1, V1  (unique)
  Layer 2: K2, V2  (unique)
  Layer 3: K3, V3  (unique)
  Layer 4: K4, V4  (unique)

CLA (sharing every 2 layers):
  Layer 1: K1, V1  (computed here)
  Layer 2: K1, V1  (reuse layer 1's cache!)   ← no compute, no extra storage
  Layer 3: K3, V3  (computed here)
  Layer 4: K3, V3  (reuse layer 3's cache!)
```

### The Intuition

Adjacent transformer layers tend to learn similar representations.
The token-level information that Layer 2 needs to attend to is often
similar to what Layer 1 already computed. Sharing exploits this redundancy.

```
Layer 1 keys (cosine similarity to Layer 2 keys): 0.87 — very similar
Layer 15 keys (similarity to Layer 16):           0.81 — still similar
Layer 30 keys (similarity to Layer 31):           0.76 — somewhat similar
```

### Cache Reduction

With CLA-2 (every 2 layers share):
- Cache reduced by ~50% vs standard MHA
- Minimal quality degradation when combined with GQA
- Models trained with CLA from scratch outperform models with CLA retrofitted

CLA is best applied during **pre-training**, not fine-tuning.

*Source: arxiv:2405.12981*

## 4. KVSharer — Similarity-Based Cross-Layer Sharing

**Paper:** arxiv:2410.18517 (2024)  
**Year:** 2024

### How It Differs from CLA

CLA uses a fixed sharing schedule (every N layers).
KVSharer uses **learned or measured similarity** to decide which layers
share, and by how much:

```
CLA:      fixed pattern: [L1=L2, L3=L4, L5=L6, ...]
KVSharer: adaptive:      [L1=L2 (sim=0.89), L3≠L4 (sim=0.42), L5=L6=L7 (sim=0.91)]
```

If two layers are very similar → share their KV cache.
If two layers are dissimilar → keep separate caches.

### When to Use

- KVSharer is better when you don't know in advance which layers are similar
- Particularly useful for fine-tuned models where layer similarity may
  differ from the base model's patterns
- Works without retraining — can be applied post-hoc with calibration data

*Source: arxiv:2410.18517*

## 5. CacheGen — KV Cache as a Streaming Format

**Paper:** arxiv:2310.07240 (2024)  
**Year:** 2024

### The Problem: KV Cache Transfer is Expensive

In disaggregated serving (Chapter 9), the KV cache must be sent
from the prefill node to the decode node. At 4K tokens, this is
~4 GB of raw tensors. Sending 4 GB over network takes seconds.

### CacheGen's Approach

Treat the KV cache like a video codec — compress it for transmission
and decompress at the receiver:

```
Prefill node:
  1. Compute KV cache (raw FP16 tensors)
  2. Compress using KV-specific codec:
     - Quantize to lower precision
     - Delta-encode between layers (adjacent layers are similar)
     - Entropy-code the result
  3. Stream compressed bytes over network

Decode node:
  1. Receive compressed stream
  2. Decompress in real-time as decode proceeds
  3. Use decompressed KV for attention
```

### Compression Ratios

CacheGen achieves 3-10x compression over raw KV tensors.
This means a 4 GB KV cache becomes 400 MB-1.3 GB — much faster to transfer.

**Key insight:** The KV cache has structure that generic codecs don't exploit:
- Adjacent layers are correlated
- Attention head outputs have bounded ranges
- Delta encoding between tokens (if sorted) is very effective

*Source: arxiv:2310.07240*

## 6. InfiniGen — Offloading-Aware Speculative KV Generation

**Paper:** arxiv:2406.19707 (2024)  
**Year:** 2024

### Context: KV Offloading to CPU

When GPU VRAM fills up, the KV cache can be offloaded to CPU DRAM
(much larger, ~512 GB typical) at the cost of slower access:

```
GPU VRAM (fast, 80 GB):   active sessions' KV cache
CPU DRAM (slow, 512 GB):  inactive sessions' KV cache, waiting

On each decode step:
  Fetch needed KV from CPU → GPU  (slow PCIe transfer: ~50 GB/s)
  Run attention                    (fast GPU compute)
```

The bottleneck becomes the PCIe bus between CPU and GPU.

### InfiniGen's Approach

**Speculative prefetching** using a lightweight draft model:

```
While generating token T on the GPU:

  Simultaneously on CPU:
    Small draft model predicts which KV blocks will be needed at token T+1
    → Prefetch those blocks from CPU DRAM to GPU VRAM in advance

  When token T finishes:
    KV blocks for T+1 are already loaded (or nearly so)
    → No waiting for CPU → GPU transfer

The key: prediction accuracy doesn't need to be 100%.
Even 70% accurate prediction covers the majority of needed blocks.
```

This overlaps computation and memory transfer — hiding the PCIe latency.

*Source: arxiv:2406.19707*

## 7. DeepSeek MLA — Multi-head Latent Attention (Deeper)

*(Introduced in Chapter 4 — key points revisited with 2024 context)*

**Papers:** DeepSeek-V2 (2024), DeepSeek-V3 (2024)

MLA compresses K and V into a **low-rank latent representation** before caching:

```
Standard GQA:
  For each token: cache K (d_h × n_kv_heads), V (d_h × n_kv_heads)
  Cache size: 2 × n_kv_heads × d_h × seq_len × bytes

MLA:
  For each token: cache compressed latent c (d_c × 1) where d_c << d_h × n_kv_heads
  At attention time: expand c → K, V using up-projection matrix (stored once)
  Cache size: d_c × seq_len × bytes  (5-13× smaller than GQA)
```

The up-projection matrix is stored as model weights (not per-token).
The per-token cache is just the tiny compressed latent.

**Why it matters:** MLA achieves GQA-level cache reduction **without** reducing
the number of attention heads or head dimension — so model quality is higher.

Used in: DeepSeek-V2, DeepSeek-V3 (2024 production models)

## 9. Open Research Problems (2025)

From the adversarially-verified research report:

**1. Eviction + Architecture Interactions**
How do SnapKV, H2O, and PyramidKV interact when combined with GQA and CLA?
If GQA already reduces KV heads by 4×, does the eviction rate assumption change?
(The optimal eviction threshold likely shifts when fewer heads exist.)

**2. Accuracy-Efficiency Frontier at Very Long Contexts (100K+)**
Do attention-sink-aware methods (StreamingLLM) consistently outperform
cumulative-score methods (H2O) at 100K+ tokens?
Current benchmarks are mostly ≤32K tokens.

**3. Disaggregated Serving + Advanced Memory Management**
How do DistServe/Mooncake interact with vAttention's CUDA VMM approach?
Does physical/virtual memory decoupling remain beneficial when prefill
and decode run on different machines entirely?

**4. Production Readiness of CLA and MLA for Fine-Tuning**
CLA and MLA are most effective when trained from scratch.
What accuracy penalty is incurred when retrofitting them to existing models?
Current evidence is limited for fine-tuning adaptation.
